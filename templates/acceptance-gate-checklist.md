# Acceptance-Gate Checklist

A reusable checklist to validate any artifact before it is accepted as complete. Parameterise per artifact type — not every check applies to every artifact.

---

## Word Count

- [ ] Word count is within the stated minimum and maximum for this artifact type.
- [ ] If no range is stated, the artifact is not egregiously long or short for its purpose.

## Required Sections

- [ ] All sections mandated by the artifact's template are present.
- [ ] No required section is empty or placeholder.
- [ ] The artifact self-declares its type (probe, packet, review, checklist, etc.).

## Changed-File Restrictions

- [ ] Only files within the artifact's intended scope were modified.
- [ ] No edits to `CANON.md` or `DECISION_LOG.md` were made, unless this task was an explicit canon or merge operation.
- [ ] If this is a branch artifact, `git diff --name-only` against the base commit does not include `CANON.md` or `DECISION_LOG.md`.

## No Canon Edits Unless Explicitly Requested

- [ ] If this task was not a canon or merge operation: confirmed that `CANON.md` and `DECISION_LOG.md` are untouched.
- [ ] If this task was a canon or merge operation: both files were updated together and logged.

## No Absolute Local Paths

- [ ] The artifact contains no absolute filesystem paths (`/Users/...`, `/home/...`, `/root/...`, etc.).
- [ ] All paths are repo-relative.

## No Local File-Scheme Links

- [ ] The artifact contains no `file:` URI scheme links.

## No Temporary Scripts Left Behind

- [ ] No untracked `.py`, `.sh`, `.js`, `.rb`, or similar scratch scripts are present in the working tree.
- [ ] `git status --short` shows only the intended changes for this artifact.

## Git Status Validation

- [ ] The working tree is in the expected state — no orphaned or uncommitted files outside the task scope.
- [ ] If the artifact was generated by an automated process, the generating script has been cleaned up.

## Branch / Non-Canon Status

- [ ] If this is a branch artifact: an explicit non-canon notice is present at or near the top.
- [ ] If this is a canon operation: both `CANON.md` and `DECISION_LOG.md` were updated and the decision log entry includes governance status (canon vs. approved guidance).

## Canon-Governance Notes

- [ ] If this is a packet, probe, or draft: all provisional specifics (names, numbers, dates, physical descriptions, staging details) are enumerated as provisional somewhere in the artifact.
- [ ] Provisional specifics are clearly marked as not promoted to canon without a separate ratification step.

## Provisional Specifics

- [ ] Every element that is invented for this artifact and not yet supported by canon is listed as provisional.
- [ ] No element listed as provisional is treated as established fact anywhere in the artifact.

## Final Verdict

- [ ] The artifact states a clear verdict or readiness status.
- [ ] If the verdict is conditional, the conditions are actionable and specific.

## Commit Readiness

- [ ] All acceptance checks pass.
- [ ] Only intended files will be staged.
- [ ] The commit message describes the artifact and its purpose.
- [ ] No uncommitted changes exist outside the intended scope.
